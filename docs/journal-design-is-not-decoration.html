What "design as decoration" actually means.

By Aarav Mehta — who designs and builds the work this essay describes.

One of the two sentences this studio publishes about the work it turns down is this: "We don't accept projects where design is treated as decoration." It's worth explaining what that actually means, because it's routinely misread as a designer's vanity — as if the objection were to clients with bad taste.

It isn't about taste. Decoration is what design becomes when it's applied at the end — when the system is already built, the product already shaped, the decisions already made, and someone is brought in to make the result look considered. At that point design can only lie. It dresses choices it had no part in, and the surface makes a promise the structure can't keep.

Design applied at the end can only describe the work. Design present from the start gets to decide it.

Treated as decision-making instead, design is the cheapest place to be right. It's where "who is this actually for" gets answered before code is written, where a workflow gets simplified before it gets automated, where a brand's one honest claim gets found before a page is laid out. Every one of those decisions costs almost nothing on paper and a great deal in production. A studio that skips them isn't saving the client money — it's deferring the bill.

This is also why the refusal protects the client, not the studio's pride. A project scoped so that design arrives last is a project scoped to rework — the expensive kind, where the change reaches back through built things. Saying no at the start is quieter and kinder than invoicing through the consequences.

The practical version of all this is simple. When we take on your work, the person thinking about how it should look is in the room when it's decided how it should work — because they're the same person. Nothing gets dressed. It gets designed, which is to say: decided.
